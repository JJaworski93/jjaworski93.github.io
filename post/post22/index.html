HTTP - methods, headers, cookies, etc.

What is an HTTP (Hypertext Transfer Protocol) request? Let's find out.

An HTTP (Hypertext Transfer Protocol) request is a message sent by a client, such as a web browser or a mobile app, to a server in order to retrieve or send information.

When client makes an HTTP request to a server, the server processes the request and sends an appropriate response back to the client. This response can be a web page, an image, a file, or an error message, among other things.

The above mentioned request includes various pieces of information, such as the method (e.g. GET, POST, PUT, DELETE), the endpoint (e.g. a specific URL), and any data or headers that are needed to complete the request.

There are several types of HTTP methods:

  • GET - to retrieve data.
  • POST - to create data.
  • PUT - to update data.
  • DELETE - to delete data.

The request also includes headers which provide additional information about the request, such as the Accept and Content-Type headers, which specify the format of the data being sent or received.

The server receives the request, processes it, and then sends back a response, which includes a status code (e.g. 200 OK, 404 Not Found) and any data or headers that are needed to complete the response.

HTTP is a stateless protocol, which means that the server does not maintain any information about the client between requests. Each request is independent of the previous one. This is the reason why cookies and sessions were introduced in the web. Cookies (and other mechanisms) can be used to maintain state between requests. It is important to test how your application handles state, especially if it relies on cookies or other stateful mechanisms.

In summary, an HTTP request is a message sent by a client to a server to initiate an action, and the HTTP response is the message sent back by the server after it processes the request.

Let’s have a look on a few common HTTP status codes and their meanings:

  • 200 OK: The request was successful and the server returned the requested data.
  • 201 Created: The request was successful and a new resource was created as a result.
  • 204 No Content: The request was successful, but there is no data to return.
  • 400 Bad Request: The request was malformed or missing required parameters.
  • 401 Unauthorized: The request requires authentication, either through a missing or invalid token.
  • 403 Forbidden: The authenticated user does not have access to the requested resource.
  • 404 Not Found: The requested resource could not be found.
  • 500 Internal Server Error: An error occurred on the server.

It is important to mention that the HTTP codes are not limited to the above codes and there are many other codes too which are specific to the context of the application. The error messages can also vary depending on the application and can be either a string or an object containing more details about the error.

It is also important to handle these errors in the client side and show the appropriate messages to the user.

Network errors. These are errors that occur when there is a problem with the network connection between the client and the server. Examples of network errors include timeouts, connectivity issues, and DNS errors. These types of errors can be caused by a variety of factors such as poor internet connection, a problem with the router, or a problem with the server.

Server errors. These are errors that occur on the server side of an application. Examples of server errors include 500 Internal Server Error, 502 Bad Gateway and 503 Service Unavailable. These errors can be caused by problems with the server’s configuration, a lack of resources, or a bug in the server-side code.

Invalid responses. These are responses that do not conform to the expected format or contain invalid data. For example, a response that is missing a required field or a response that contains an unexpected data type. These types of errors can be caused by bugs in the server-side code or by problems with the client’s request.

It is worth noting that different types of errors may require different types of testing and different tools. Network errors can be tested by simulating network conditions and measuring the application’s behavior. Server errors can be tested by creating test cases that simulate different types of server errors and verifying that the application behaves correctly. Invalid responses can be tested by sending requests with different types of invalid data and verifying that the application returns the correct error messages.

You must have heard the terms “HTTP call” and “HTTP request” in the past. Those are often used interchangeably to refer to the process of sending an HTTP message from a client to a server. However, there is a subtle difference between the two terms.

An HTTP call refers to the act of initiating an HTTP request and receiving an HTTP response. It is a higher-level concept that encompasses both the request and the response, and it can include additional steps such as parsing the response and handling errors.

An HTTP request, on the other hand, is a specific message sent by a client to a server. It includes a method (e.g. GET, POST), an endpoint (e.g. a specific URL), headers, and a body (if applicable).

An HTTP response is the message sent back by the server in response to an HTTP request. It includes a status code, headers, and a body (if applicable).

So, in short, an HTTP call refers to the whole process of requesting and receiving a response, while an HTTP request refers to the specific message sent by the client to the server.

An HTTP header is a piece of information that is included in an HTTP request or response, and is used to provide additional context about the message. HTTP headers are key-value pairs, with the key being the name of the header and the value being its associated data.

HTTP headers can be used to provide various types of information, such as:

  • Request headers:
    • Accept: Specifies the format of the data that the client can handle.
    • Content-Type: Specifies the format of the data in the request body.
    • Authorization: Provides authentication information for the request.
    • User-Agent: Identifies the client software and version.
  • Response headers:
    • Content-Type: Specifies the format of the data in the response body.
    • Content-Length: Specifies the size of the response body.
    • Server: Identifies the server software and version.
    • Location: Specifies the URL of the resource that the client should access next.

Headers are used by the server to understand the request and by the client to understand the response. They can also be used for security, for example, the server can set a cookie in the Set-Cookie header of the response, that the client will have to use in the next requests in the Cookie header.

It’s important to note that header fields are case-insensitive, headers fields names are separated from their values by a colon, and headers fields can span multiple lines if the value contains whitespace.

You must also have heard about Cookies. Cookies are small text files that are stored on a client’s computer by a web server when the client visits a website. They are used to remember certain information about the user, such as their preferences or login credentials.

Cookies are sent by the server to the client in the form of an HTTP header called “Set-Cookie” in the response. The browser then stores the cookie and returns it to the server in the “Cookie” header of subsequent requests.

Cookies can be used for various purposes, such as:

  • Storing user preferences, such as language settings or theme selection.
  • Keeping track of the user’s session, so the user doesn’t have to re-enter their login credentials each time they visit the website.
  • Storing shopping cart information for e-commerce websites.
  • Tracking user behavior for analytics or advertising purposes.

It is important to note that cookies are stored on the client-side, and they can be modified, deleted or blocked by the client.

Cookies also have an expiration date, which can be set by the server, after that date the browser will delete the cookie. It is important to test that our application sets the correct expiration date, and that it behaves correctly when a cookie has expired.

Cookie Scope. Cookies have a scope, which determines which pages on a website can access them. It is important to test that our application sets the correct scope for each cookie, and that it behaves correctly when a cookie is not accessible.

Cookie Security. Cookies can be secured to prevent them from being accessed by malicious actors. It is important to test that our application sets the correct security settings for each cookie, and that it behaves correctly when a cookie is not secure.

Additionally, cookies can be marked as “secure” and “HTTPOnly” which means that they will only be sent over a secure connection (HTTPS) and they can not be accessed by client-side scripts (javascript).

Cookies are a simple and efficient way to store user data, but they can also pose security risks if they are not used properly.

Overall, testing HTTP calls and cookies requires a combination of functional, performance and security testing, it’s important to understand the behavior of cookies and how it affects the overall application and its interactions.

Caching is the process of storing frequently accessed data in a temporary storage area, such as memory, so that it can be quickly retrieved and used again without having to be recomputed or retrieved from the original data source.

In the context of HTTP, caching can refer to the temporary storage of HTTP responses on a client (e.g. a web browser) or a server. This allows subsequent requests for the same resource to be served more quickly, as the data is already available in the cache. Caching can improve the performance of an application by reducing the number of round-trips to the server and by reducing the amount of data that needs to be transmitted over the network. Browsers can cache HTTP responses, including cookies.

Caching can be controlled by various headers in the HTTP response, such as the “Cache-Control” header, which specifies how long the response can be cached and under what conditions it can be reused. When making an HTTP request, the client may also include headers such as “If-Modified-Since” or “If-None-Match” to indicate that it has a cached version of the resource and only wants to retrieve it again if it has changed on the server. t’s important to test that your application sets the correct caching headers to prevent caching when it should not be cached and also to test how the application behaves when it receives a cached response.
