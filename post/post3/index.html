The differences between TDD and BDD

Let's look at the differences between two testing-related concepts - Test Driven Development and Behavior Driven Development.

In this blog post we are going to discover the main differences between two testing-related concepts - Test Driven Development and Behavior Driven Development.

TDD and BDD are two widely used software development techniques in which automated testing plays a central role. Today we are going to explore the differences.

Let’s start with Test Driven Development. Typically, in this technique, we write tests for a certain piece of functionality before writing the production code. Then, we run those tests, we wait for them to fail and then we write a code to make the tests pass. This will give us the confidence that our code works. In general, we can say that TDD is all about Unit Testing. This approach is mainly focused on achieving high test coverage and answering following questions - is the code working? Was the code/feature implemented? Steps here will be:

  • write a test case based on requirements
  • write an unit test for the piece of the codebase (functionality) you are planning to create,
  • execute all existing tests (if any exist) and make sure that only the newly created test is failing. It must fail, as you try to test something that doesn’t yet exist,
  • write production code to make all the tests pass,
  • refactor your code in order to improve it in general (e.g. it’s maintainability, extensibility, readability, efficiency etc.) and to avoid things like duplication,
  • run all the tests again, to be sure that refactoring didn’t break anything.

When it comes to Behavior Driven Development, usually, it involves developers, test/qa engineers, product managers, business analysts and potentially also other stakeholders. Their task is to discuss the concrete examples of so-called acceptance criteria in a user story. In short, it can be said that the acceptance criteria is a set of conditions, as well as an instruction on how software should function, look and behave to meet the user’s needs. To simplify, we can say that this technique is focused on writing acceptance criteria by giving examples of how the software should behave in a certain scenarios. This makes it easier for all stakeholders to collaborate. Most companies use the Given-When-Then approach for this. Let’s see an example:

  • Given - user has provided valid login credentials,
  • When - user has clicked the login button,
  • Then - user gets a message about successful login.

BDD is a great automated testing practice, as it focuses on customer experience and not only on code/feature implementation. Steps here will be:

  • identify and describe new feature and it’s behavior under certain scenarios,
  • write understandable for everybody steps for each scenario,
  • write tests based on those scenarios,
  • implement the respective code,
  • run tests and check if they are successful. If not, fix the errors. If yes, move further,
  • refactor your code,
  • run all the tests again.

To summarize, we can say that TDD is focused more on unit testing, code/feature implementation and high test coverage, while BDD is focused more on user experience and software behavior.
